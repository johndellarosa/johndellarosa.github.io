Recommended Software

An alternative to ctypes is the Python C API, which allows you to write C extensions that can be imported directly into Python. This is more complex but offers better performance and integration with Python's memory management.

Anatomy of a C Extension

PyMODINIT_FUNC

PyMODINIT_FUNC is a macro used to declare the module initialization function which CPython calls when it loads an extension module, returning a PyObject * on success and NULL on failure. The basic structure is as follows: 

                    PyMODINIT_FUNC PyInit_mymod(void) {
                        /* ... create & initialize a module object ... */
                        return module;  // PyObject*
                    }
Note: The name must be PyInit_, where is the module's build name (the filename without the extension suffix).

Import Lifecycle

When you import an extension module, CPython's dynamic loader:
  1. Loads the shared library (.so, .pyd, .dylib)
  2. Looks up the symbol PyInit_<basename>
  3. Calls it
  4. If it returns a non-NULL PyObject *, that object becomes the module placed in sys.modules. If it returns NULL, the import fails and the current exception is raised.
If you have an extension compiled as C++, extern "C" stops name mangling which can prevent the loader from finding the PyMODINIT_FUNC. 

                    extern "C"
                    PyMODINIT_FUNC PyInit_mymod(void) {
                        return PyModuleDef_Init(&moddef);
                    }

Reload

For multi-phase modules, CPython re-executes your Py_mod_exec for the (re)loaded module. You must write exec to be idempotent (i.e., not leak or double-register things). Several tips:
  • Recreate objects you own each time and re-add them to the module
  • If you keep references in state, either
    • Clear previous ones in m_clear (reload triggers clear before exec), or
    • Guard against double-init (check st->Error == NULL etc.).
